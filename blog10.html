Kubernetes Part 3-After Deployment

Last week I covered a little bit about creating kubernetes clusters and deployments on blog 9. This time we will explore a little bit more about the kubernetes pods and nodes.


What is a Kubernetes pod?


A kubernetes pod is a group of containers that are deployed together on the same host. With that being said, all containers inside this pod are sharing some resources. All these containers within a single pod share the same storage, network, and also specifications on how to run the containers.


A pod is an atomic unit on kubernetes. Whenever we create a deployment, pods are created with containers inside them. Each Pod is tied to the Node where it is scheduled. It will remain inside the node until it is terminated or deleted. If there's a failure on the node, identical pods will be created and will be placed in other available nodes within the cluster.


Kubernetes Node


A pod always run on a node. As we recall, a node is the workers that runs the applications. It is a VM or a physical computer that serves as a worker machine in a Kubernetes cluster.A Node can have multiple pods, and the Kubernetes master automatically handles scheduling the pods across the Nodes in the cluster.


Each node would run:

  • kubelet: this is a process that handles the communications between the kubernetes master and the node. A kubelet manages pods and containers running on a machine.
  • container runtime: responsibles for pulling the container image from a registry, unpacking the container, and running the application.


Some useful kubectl commands related to nodes and pods:

  • kubectl get --> to list all resources
  • kubectl describe --> this command is used to show detailed information about a resource.
  • kubectl logs --> this command will print the logs from a container in a pod.
  • kubectl exec --> this command is used to execute a command on a container in a pod
